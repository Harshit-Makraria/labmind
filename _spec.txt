1. Project Overview
LabMind is a mobile-first agentic AI companion for physical science labs. It watches, guides, and corrects students in real time during live experiments — replacing the absent TA with a multimodal AI agent that reads burette levels from photos, cross-checks experimental steps, detects reagent conflicts, and delivers experiment-specific result interpretation.

2. Complete Feature Specifications
2.1  Feature 1 — Protocol Ingestion &amp; Step-by-Step Guidance
Description
Student uploads a lab PDF. The backend parses it into an ordered state machine of steps. Each step presents: reagents needed, timing, safety flags, and the underlying science explanation. A photo capture is student-initiated at step completion.

Functional Requirements
PDF upload UI on the student home screen — accept .pdf files up to 10 MB
On upload, call POST /api/protocol/parse — backend sends PDF text to GPT-4o to extract steps as a JSON state machine
Each step object contains: step_number, title, instructions[], reagents[], duration_seconds, safety_flags[], science_explanation
Render step cards one at a time: large step number, instruction text, reagent chips, a timer (if duration specified), safety flag banner (red if safety_flags non-empty)
&apos;Mark Complete &amp; Capture Photo&apos; button at bottom of each step card — opens camera/file picker
After photo, trigger Feature 2 (Vision Check) before advancing to next step
Cross-step reasoning: if a step is skipped or failed, flag dependent downstream steps with a yellow warning badge
Progress bar at top shows completed / total steps

State Machine Schema (JSON)
POST /api/protocol/parse
Request:  { &quot;pdf_base64&quot;: &quot;&lt;base64 string&gt;&quot;, &quot;session_id&quot;: &quot;uuid&quot; }
Response: {
  &quot;experiment_name&quot;: &quot;Acid-Base Titration&quot;,
  &quot;steps&quot;: [
    {
      &quot;step_number&quot;: 1,
      &quot;title&quot;: &quot;Prepare Burette&quot;,
      &quot;instructions&quot;: [&quot;Rinse burette with distilled water&quot;, &quot;Fill with 0.1M NaOH&quot;],
      &quot;reagents&quot;: [{ &quot;name&quot;: &quot;NaOH&quot;, &quot;concentration&quot;: &quot;0.1M&quot;, &quot;volume_ml&quot;: 50 }],
      &quot;duration_seconds&quot;: 300,
      &quot;safety_flags&quot;: [],
      &quot;science_explanation&quot;: &quot;Rinsing removes contaminants that alter molarity.&quot;,
      &quot;expected_observation&quot;: &quot;Burette reads 0.00 mL at start&quot;,
      &quot;vision_check_required&quot;: true,
      &quot;vision_expected&quot;: { &quot;type&quot;: &quot;burette_reading&quot;, &quot;expected_value&quot;: 0.0, &quot;tolerance&quot;: 0.1 }
    }
  ]
}

2.2  Feature 2 — Computer Vision Photo Verification
Description
When a student taps &apos;Capture Photo&apos; after completing a step, the image is sent to GPT-4o Vision. The model reads the burette level (or colour change, or gel band — experiment-dependent) and compares to the expected observation. Low-confidence results trigger a re-photograph prompt before the student can proceed.

Functional Requirements
Student captures a photo using the device camera (mobile-first) or uploads an image file
Image sent to POST /api/vision/check along with current step context
Backend calls GPT-4o Vision with a structured prompt: &apos;Read the burette meniscus value in this photo. Respond with JSON: { reading: float, confidence: float 0-1, notes: string }&apos;
If confidence &gt;= 0.75 AND reading within tolerance: show green check, allow proceed
If confidence &lt; 0.75 OR reading out of tolerance: show amber warning with specific message, prompt re-photograph
After 2 failed re-photographs: allow manual override with student-entered value + note logged for instructor
Vision result stored in session state for Result Interpretation (Feature 4)
Streaming response from GPT-4o — show typing indicator, target &lt; 3s perceived latency

API Contract
POST /api/vision/check
Request: {
  &quot;session_id&quot;: &quot;uuid&quot;,
  &quot;step_number&quot;: 4,
  &quot;image_base64&quot;: &quot;&lt;base64&gt;&quot;,
  &quot;expected&quot;: { &quot;type&quot;: &quot;burette_reading&quot;, &quot;expected_value&quot;: 24.5, &quot;tolerance&quot;: 0.1 }
}
Response: {
  &quot;reading&quot;: 24.6,
  &quot;confidence&quot;: 0.91,
  &quot;pass&quot;: true,
  &quot;deviation&quot;: 0.1,
  &quot;message&quot;: &quot;Burette reads 24.60 mL — within tolerance (±0.1 mL). Proceed.&quot;,
  &quot;notes&quot;: &quot;Meniscus clearly visible, lighting good.&quot;
}

2.3  Feature 3 — Safety Alert Engine
Description
Reagent conflict detection runs on every step. If a step introduces a reagent that conflicts with a reagent already used (e.g. mixing HCl and NaOH outside titration context, or adding concentrated acid to water in wrong order), an immediate modal alert is displayed that halts progress until the student acknowledges the risk.

Functional Requirements
Hardcode a reagent safety database (JSON file in backend) for Phase 1 with at least 20 reagent pairs and their conflict type/severity
On each step advance, call POST /api/safety/check with the step&apos;s reagents + session reagent history
If conflict detected: display full-screen red modal with: reagent names, conflict type, hazard description, recommended action, cannot be dismissed without pressing &apos;I Acknowledge the Risk&apos;
Log all safety alerts to session record (visible on instructor dashboard)
Safety check is synchronous — student cannot proceed until check completes
For prototype: also check concentration (e.g., &gt; 6M HCl → strong acid warning)

Safety Database Sample (reagent_safety.json)
{
  &quot;conflicts&quot;: [
    { &quot;reagents&quot;: [&quot;HCl&quot;, &quot;NaOH&quot;], &quot;type&quot;: &quot;Neutralization&quot;, &quot;severity&quot;: &quot;medium&quot;,
      &quot;description&quot;: &quot;Exothermic neutralization. Controlled in titration context.&quot;,
      &quot;action&quot;: &quot;Proceed slowly. Swirl flask gently.&quot; },
    { &quot;reagents&quot;: [&quot;H2SO4&quot;, &quot;NaOH&quot;], &quot;type&quot;: &quot;Violent Neutralization&quot;, &quot;severity&quot;: &quot;high&quot;,
      &quot;description&quot;: &quot;Concentrated H2SO4 + NaOH: extreme exothermic reaction.&quot;,
      &quot;action&quot;: &quot;STOP. Verify concentrations. Use dilute solutions only.&quot; },
    { &quot;reagents&quot;: [&quot;KMnO4&quot;, &quot;H2O2&quot;], &quot;type&quot;: &quot;Oxidizer Conflict&quot;, &quot;severity&quot;: &quot;high&quot;,
      &quot;description&quot;: &quot;KMnO4 acts as oxidizer; H2O2 as reductant — vigorous decomposition.&quot;,
      &quot;action&quot;: &quot;Do not combine. Contact instructor immediately.&quot; }
  ]
}

2.4  Feature 4 — Result Interpretation Engine
Description
After the final step, the student enters their calculated result (e.g. concentration of unknown acid). The engine computes % deviation from the theoretical value, diagnoses likely error sources based on the specific deviation pattern, and presents a structured learning output.

Functional Requirements
Final screen: &apos;Enter Your Result&apos; — numeric input field + unit label
On submit: call POST /api/results/interpret
Backend computes: % deviation = |student_result - theoretical| / theoretical * 100
GPT-4o generates experiment-specific diagnosis using the session&apos;s vision readings and step history as context
Example diagnosis logic in prompt: &apos;If deviation 8-12%: likely parallax error at meniscus. If deviation &gt; 15%: likely reagent contamination or concentration error. If negative deviation: endpoint overshot.&apos;
Display result card with: student value, theoretical value, % deviation (green &lt;2%, amber 2-10%, red &gt;10%), diagnosis text, suggested improvement for next trial
Instructor dashboard updates with student&apos;s result and deviation

API Contract
POST /api/results/interpret
Request: {
  &quot;session_id&quot;: &quot;uuid&quot;,
  &quot;student_result&quot;: 0.094,
  &quot;unit&quot;: &quot;mol/L&quot;,
  &quot;theoretical_value&quot;: 0.1,
  &quot;session_history&quot;: { &quot;steps_completed&quot;: 8, &quot;vision_readings&quot;: [...], &quot;safety_alerts&quot;: [] }
}
Response: {
  &quot;deviation_percent&quot;: 6.0,
  &quot;severity&quot;: &quot;amber&quot;,
  &quot;diagnosis&quot;: &quot;6.0% deviation suggests a likely parallax error when reading the burette meniscus at step 4. Your vision check showed 24.6 mL vs expected 24.5 mL — the cumulative 0.1 mL offset over multiple readings compounds to this deviation.&quot;,
  &quot;improvement&quot;: &quot;Always read the burette at eye level. Use a white card behind the burette to improve meniscus contrast.&quot;,
  &quot;learning_point&quot;: &quot;Parallax error is systematic — it shifts all readings in the same direction. If your result is consistently low, check your reading angle.&quot;
}

2.5  Feature 5 — Instructor Dashboard
Description
A separate web view for the instructor (accessed via /dashboard) shows class-wide real-time progress for all active sessions. The instructor can see which step each student is on, any safety alerts triggered, vision check failures, and final result deviations.

Functional Requirements
Route: GET /dashboard — protected by a simple instructor passcode (hardcode: &apos;labmind2026&apos; for prototype)
Shows a grid of student session cards, each with: session ID / student name, current step number out of total, status indicator (active/completed/safety alert), last vision check result
Real-time updates via WebSocket (ws://localhost:8000/ws/dashboard) — server pushes step_complete, safety_alert, result_submitted events
Safety alert cards highlighted in red with flashing badge
Summary bar at top: total active sessions, average completion %, number of safety alerts today
Clicking a student card expands to show full step history and all vision readings
Export button: download CSV of all session results for the class


3. System Architecture
3.1  Architecture Overview
LabMind follows a clean client-server architecture with an agentic orchestration layer. The Semantic Kernel orchestrator owns the state machine and agent loop. LangChain provides domain-specific tool implementations. FastAPI serves the REST and WebSocket API. The React PWA is the student-facing frontend.

Tech Stack (Capgemini Official Stack Aligned)
LLM + Vision: Azure OpenAI (GPT-4o)  |  Agent Orchestration: Semantic Kernel + LangChain  |  Backend: Python FastAPI  |  Frontend: React PWA (Tailwind CSS)  |  Observability: OpenTelemetry + Python logging  |  Testing: Golden dataset + pytest

3.4  Agentic Loop Detail (Semantic Kernel)
The Semantic Kernel orchestrator manages a session state object and selects LangChain tools based on the current state. This is the core &apos;agentic&apos; behaviour described in the pitch deck:
State: { session_id, experiment_name, steps[], current_step_index, session_history, status }
On each student action, the orchestrator evaluates: what tool is needed? (parse / safety / vision / interpret)
Tool selection is deterministic for the prototype (rule-based, not fully autonomous) — this is appropriate for a safety-critical domain
Failure mode: if any tool call fails after 2 retries → orchestrator returns a step-by-step fallback instruction prompt to keep the student moving safely
All tool invocations logged via OpenTelemetry spans with: tool_name, input_summary, output_summary, latency_ms, confidence score

6.2  GPT-4o Vision Prompt (vision_tool.py)
VISION_PROMPT = &apos;&apos;&apos;
You are a chemistry lab assistant. Analyse the provided image of a laboratory burette.

Task: Read the liquid level (meniscus) in the burette.

Rules:
- Read the BOTTOM of the meniscus for clear/light-coloured liquids
- Read the TOP of the meniscus for dark liquids (e.g. KMnO4)
- Burettes read top-to-bottom (0 at top, 50 at bottom)
- Report to 2 decimal places

Respond ONLY with valid JSON (no markdown):
{ &quot;reading&quot;: &lt;float&gt;, &quot;confidence&quot;: &lt;float 0-1&gt;, &quot;notes&quot;: &quot;&lt;string&gt;&quot; }

If no burette is visible, return: { &quot;reading&quot;: null, &quot;confidence&quot;: 0, &quot;notes&quot;: &quot;No burette detected&quot; }

6.3  Protocol Parser Prompt
PROTOCOL_PARSE_PROMPT = &apos;&apos;&apos;
You are a chemistry lab protocol parser. Given the following lab manual text, extract
a structured step-by-step protocol as JSON.

For each step, extract:
- step_number (int)
- title (string, max 8 words)
- instructions (list of strings, each a single action)
- reagents (list of {name, concentration, volume_ml} — omit unknown fields)
- duration_seconds (int or null)
- safety_flags (list of strings — any safety concerns mentioned)
- science_explanation (1-2 sentences explaining the chemistry of this step)
- expected_observation (string — what the student should see)
- vision_check_required (bool — true if a burette/colour change reading is needed)
- vision_expected (object or null)

Respond ONLY with valid JSON: { &quot;experiment_name&quot;: &quot;...&quot;, &quot;steps&quot;: [...] }
&apos;&apos;&apos;

6.4  WebSocket Dashboard Implementation
# backend/routers/dashboard.py
from fastapi import WebSocket, WebSocketDisconnect
from agents.session_manager import session_store

active_connections: list[WebSocket] = []

async def broadcast(event: dict):
    for ws in active_connections:
        await ws.send_json(event)

@router.websocket(&apos;/ws/dashboard&apos;)
async def dashboard_ws(websocket: WebSocket):
    await websocket.accept()
    active_connections.append(websocket)
    try:
        while True:
            await websocket.receive_text()  # keep alive
    except WebSocketDisconnect:
        active_connections.remove(websocket)
